# Storage Architecture Memory

## Non-negotiable rule

`localStorage` and `sessionStorage` are forbidden for application data. This is a zero-acceptability constraint and applies to all new code, migrations, fallbacks, tests, and UI features. Do not introduce a browser key/value persistence layer, even for small or temporary records.

## Canonical architecture

- Application records are local-first and belong in IndexedDB through `src/lib/storage/persistent.ts` and the shared `Storage` implementation.
- IndexedDB database: `kenzie-deskclerk`.
- IndexedDB stores include `projects`, `notes`, `tasks`, `documents`, `time_logs`, `launchpad_links`, `events`, and `settings`.
- Use `PersistentCollection` for collection CRUD and the typed models in `src/lib/storage/models.ts` for record shape.
- Metadata IDs are generated by the application/storage boundary. AI-proposed create payloads must never control IDs; discard any incoming `id` before creation.
- Application state may be represented with Svelte stores, but persistence must go through the storage abstraction rather than directly through browser globals.

## Required implementation pattern

1. Add or use a typed IndexedDB store/model.
2. Load and mutate records through `PersistentCollection` or a dedicated typed repository.
3. Keep Svelte stores synchronized with the repository result.
4. Add unit tests for persistence and reload behavior, plus e2e coverage where the feature is user-visible.
5. For Tauri/native environments, keep the same repository contract so the backend can later be swapped without changing UI code.

## Current violations to remove

The repository currently contains legacy direct browser persistence in Clerk state, documents/Quicknotes, routines, time logs, and SearchSidebar. These are technical debt, not approved architecture. New work must not copy these patterns. Migrations should move each record type into IndexedDB and then remove the old browser-storage code and tests.

## Verification guidance

Search before submitting storage-related work:

```text
rg "localStorage|sessionStorage" src tests
```

Any new production match is a regression. Tests may use an in-memory fake repository or IndexedDB test double; they must not normalize direct `localStorage` usage in application code.
